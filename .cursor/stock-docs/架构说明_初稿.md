# Flow2Spec（@double-codeing/flow2spec）架构说明（初稿）

> **说明**：本初稿由**扫描仓库目录与关键源码**归纳而成，未结合未入库或未读的业务背景；建议按实际演进补充边界、发布流程与协作约定。若需规范成终稿格式，可再使用 **f2s-doc-final**，随后用 **f2s-ctx-build** 生成 Rules / Skills。

---

## 1. 项目定位

**Flow2Spec** 是一个 **Node.js 命令行 npm 包**，对外提供可执行入口 **`flow2spec`**（见 `package.json` 的 `bin` 与 `main`）。核心能力是：在**目标业务仓库**（「配置根」的父目录）执行 **`flow2spec init`**，把包内 **`templates/`** 下的 **Rules、Skills、文档模版与预建目录** 复制到用户选择的 **AI 工具配置根**（如 `.cursor/`、`.claude/`、`.codex/`），从而落盘一套**文档驱动、可日复一日的 f2s 工作流**与知识库目录结构。

包自身**不是**业务应用框架；**运行时职责**主要是 **CLI 解析 + 文件系统复制**，无服务端、无持久化数据库。

## 2. 技术栈与依赖

| 项 | 内容 |
|----|------|
| 运行时 | Node.js **≥ 16**（`engines`） |
| 语言与模块 | **CommonJS**（`require` / `module.exports`） |
| 运行时依赖 | `package.json` 中**未声明** `dependencies`，init 逻辑仅使用 **Node 内置** `fs`、`path` |
| 分发 | npm 包名 `@double-codeing/flow2spec`；`files` 字段限定发布：`cli.js`、`lib/`、`templates/`、`README.md`、`docs/` |

---

## 3. 仓库顶层结构（与职责）

| 路径 | 职责（归纳） |
|------|----------------|
| **`cli.js`** | 进程入口：解析 `argv`；支持 `--help` / `init [agent ...]`；打印帮助文案并调用 `lib/init` |
| **`lib/agents.js`** | 声明支持的 **agent → 配置根目录** 映射（`cursor` / `claude` / `codex`）、`SUBDIRS` 列表、`normalizeAgentIds`（默认 `cursor`、去重、校验未知 agent） |
| **`lib/init.js`** | **`runInit`**：为每个选中 agent 创建 `stock-docs`、`req-docs`、`template`、`rules`、`skills`；将 `templates/` 下 `rules`、`skills`、`template` **递归复制**到对应配置根（已存在则覆盖） |
| **`templates/`** | **权威模板源**：随包发布；init 时复制到用户仓库的配置根内 |
| **`docs/`** | 使用说明、路径约定、命令说明、原理与配图等**人类可读文档**（与 README 交叉引用） |
| **`README.md`** | 产品级总览：闭环理念、快速开始、能力表与文档导航 |
| **`.cursor/`**（本仓库内） | 开发/自举时的 **Cursor 配置根**示例：含已展开的 `rules`、`skills`、`stock-docs`、`req-docs` 等（与 `templates/` 可能不同步，以各自用途为准） |
| **`scripts/`** | 当前仓库中目录存在；未发现脚本文件（可能占位或待扩展） |

---

## 4. 运行流程（CLI → init）

1. 用户在**业务仓库根目录**执行 `npx @double-codeing/flow2spec init` 或全局安装后的 `flow2spec init`。
2. **`cli.js`** 将 `init` 后的参数作为 agent 列表传入 **`lib/init.run(cwd, agentIds)`**；`cwd` 为 `process.cwd()`。
3. **`normalizeAgentIds`**：无参数时等价于 `['cursor']`；支持多个 agent、去重、拒绝未知 id。
4. 对每个 agent：**`ensureDirs`** 在 `cwd/<agentRoot>/` 下创建 `SUBDIRS`；**`copyTemplatesToAgentRoot`** 从包内 `templates/` 同步 `rules`、`skills`、`template` 到该配置根。

**边界**：init **不**修改业务源码目录结构（除在配置根下写入/覆盖模板文件外）；**不**负责 Cursor/Claude/Codex 插件侧的实际加载行为，仅提供约定目录与内容。

---

## 5. 模板与配置根约定（概念层）

- **配置根**：各 AI 工具约定的项目内目录（如 `.cursor`），其下通常包含 **`rules/`**、**`skills/`**（各子目录含 `SKILL.md`）、**`template/`**、以及预建的 **`stock-docs/`**（存量/架构类文档）、**`req-docs/`**（需求与按实现演进的技术方案）等。
- **`templates/skills`**：以技能标识为子目录名，承载 **f2s-*** 与配套工作流说明（与 README 中的能力表一致）。
- **`templates/rules`**：如 `implement-tech-design.mdc`、`stock-docs-vs-req-docs.mdc` 等专题规则。
- **`templates/template`**：如终稿模版、后端技术模版等 Markdown 模版。

（具体文件清单会随版本迭代；以 `templates/` 与发布包内实际树为准。）

## 6. 测试与质量门禁

- **`npm test`** / **`prepublishOnly`**：执行 `node cli.js --help`，作为最小冒烟，确保 CLI 可启动且帮助输出正常。
- **`pack:check`**：`npm pack --dry-run` 校验发布包内容是否符合 `files` 约定。

## 7. 与「使用方仓库」的关系

| 角色 | 说明 |
|------|------|
| **本包仓库** | 维护 CLI、`lib`、**`templates/`** 与 **`docs/`**；版本号在 `package.json` |
| **消费者仓库** | 通过 init 获得**配置根**下的规则、技能与目录；日常在对话中按 Skill 触发 **f2s-*** 工作流，与业务代码并行演进 |
