# 命令说明

本文档说明 **Flow2Spec** 各命令的**入参、输出与行为**，并按**使用顺序**提供速查。文中 **「配置根」** 指 `flow2spec init` 所选 AI 工具目录（默认 **`.cursor/`**，亦可是 **`.claude/`**、**`.codex/`** 等）；子路径（`stock-docs/`、`rules/` 等）结构相同。目录与路径约定见 [README-目录与路径约定](./README-目录与路径约定.md)（§1 目录结构）；init 详解见 [Flow2Spec使用说明](./Flow2Spec使用说明.md#一init-做了什么)；架构见 [README-体系与原理](./README-体系与原理.md)。

## 按使用顺序查找

按**推荐使用顺序**列出的命令速查，便于按流程直接跳转到对应说明。

| 阶段             | 步骤            | 命令                              | 一句话说明                                                               | 详见                                             |
| ---------------- | --------------- | --------------------------------- | ------------------------------------------------------------------------ | ------------------------------------------------ |
| **项目首次使用** | —               | `flow2spec init [agent ...]`      | 安装 OpenSpec；按所选 agent 写入**配置根**模板与 **openspec/**（详见 `--help`） | [§1](#1-flow2spec-initcli)                       |
| **需求与方案**   | 1               | **/requirement-clarify**                     | 针对 PRD/需求反问直到需求完全清楚                                        | [§2.0.1](#201-requirement-clarify)                          |
|                  | 2               | **/gen-backend-tech-doc**             | 根据澄清后需求基于项目知识库生成后端技术文档                             | [§2.0.2](#202-gen-backend-tech-doc)                          |

|                  | 2               | **/spec2context-md**              | 将初稿/PDF/MD 转为《终稿模版》**终稿**                                   | [§2.2](#22-spec2context-md)                      |
|                  | 3               | **/generateProjectContext**       | 根据终稿生成 **Rules、Skills、文档索引**                                 | [§2.3](#23-generateprojectcontext)               |
|                  | 配合            | **/deleteProjectContext**         | 删除某文档对应的 Rules、Skills、索引                                     | [§2.4](#24-deleteprojectcontext)                 |
|                  | 可选            | **/pdf4code-md**                  | 仅有 PDF 时先转成 MD                                                     | [§2.5](#25-pdf4code-md)                          |
| **提问与实现**   | 1               | **/opsx-new**                     | 提问并新建变更                                                           | [§3.1](#31-openspec-工作流)                      |
|                  | 2               | **/opsx-continue** 、**/opsx-ff** | 根据提案生成 specs、design、tasks                                        | [§3.1](#31-openspec-工作流)                      |
|                  | 3               | **/opsx-apply**                   | 根据 tasks 生成代码                                                      | [§3.1](#31-openspec-工作流)                      |
| **实现后**       | 按需            | **/global-fix**                   | 用户指出规则错误时：改代码并同步文档与规则                               | [§3.2](#32-global-fix--global-feat--global-sync) |
|                  | 按需            | **/global-feat**                  | 新增能力：补全实现与文档，或仅补文档                                     | [§3.2](#32-global-fix--global-feat--global-sync) |
|                  | 可选            | **/global-sync**                  | 方案→概述→全局 Rules/Skills→同步规范                                     | [§3.2](#32-global-fix--global-feat--global-sync) |
|                  | merge/rebase 后 | **/global-merge-context**         | 解决冲突标记：索引/规则/技能等上下文自动合并；实现与配置类仅列差异待确认 | [§3.3](#33-global-merge-context)                 |

**汇总顺序**：init → 需求与方案（需求澄清 → 后端方案）→ 上下文生成（genStructureDoc → spec2context-md → generateProjectContext）→ 提问与实现（可选 pdf4code-md → opsx-new → opsx-continue → opsx-apply）→ 实现后（global-fix / global-feat / global-sync）；**merge/rebase 出现 `<<<<<<<` 等冲突时**可用 **global-merge-context**（与上述无固定先后）。详见 [§4 推荐执行顺序](#4-推荐执行顺序flow2spec)。

## 1. flow2spec init（CLI）

在**配置根父目录**执行，安装 OpenSpec（若未安装），并按所选 **agent** 将模板写入对应**配置根**；**openspec/** 始终复制到**配置根父目录**（一份）。

| 项目          | 说明                                                                                                                                                                                                                                                   |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **用法**      | `npx @lands/flow2spec init` 或 `flow2spec init [cursor \| claude \| codex ...]`（全局安装后）；省略 agent 时默认为 **cursor**（写入 **`.cursor/`**）。完整列表与示例见 **`flow2spec --help`**。                                                          |
| **创建/写入** | 对每个所选 agent：在配置根下创建 `stock-docs/`、`req-docs/`、`template/`、`rules/`、`skills/`、`commands/` 并复制模板；**openspec/** 复制到**配置根父目录**。详见 [Flow2Spec使用说明 - init 做了什么](./Flow2Spec使用说明.md#一init-做了什么)。 |
| **结果**      | 配置根为 **`.cursor/`** 时，在 Cursor 中输入 `/` 可使用斜杠命令；可与 OpenSpec CLI 配合使用 **openspec/**。                                                                                                                                                              |

## 2. 阶段一：需求与方案、上下文生成

### 2.0 需求与方案（可选）

当用户手头有 PRD 或需求描述、希望先澄清再产出后端技术方案时，可先使用 **/需求澄清**，澄清完成后使用 **/生成后端技术文档**。

#### 2.0.1 /requirement-clarify

针对 PRD 或需求中的模糊、缺口、歧义进行反问，直到需求足够清晰。

| 项目     | 说明                                                                                                                                                                     |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **入参** | 可选。**第一参数**：PRD 全文、需求描述，或**文档路径**（如 **`.cursor/req-docs/xxx.md`**）；不传则基于当前对话中用户已发内容进行澄清。用户可在后续回复中补充需求条件或约束。     |
| **行为** | 识别需求中的模糊表述、未定义概念、缺失信息、潜在矛盾及与实现强相关但未说明的点 → 分组、具体可答地反问 → 根据用户回答迭代追问，直到主要流程、边界、异常、关键概念无歧义。 |
| **输出** | 无固定产出；澄清结束时可以简要总结「当前需求要点」，并提示用户可使用 **/生成后端技术文档** 生成技术文档。                                                               |

**注意**：本命令仅做需求澄清，不在此阶段输出技术方案或接口设计。

#### 2.0.2 /gen-backend-tech-doc

根据**已澄清的需求**（及可选需求条件），基于当前项目的**知识库、`skills/`、`rules/`**（Cursor 下即 `.cursor/skills`、`.cursor/rules`）生成针对性的**后端技术文档**。

| 项目         | 说明                                                                                                                                                                          |
| ------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **入参**     | **第一参数（必填）**：澄清后的需求描述或需求/PRD 文档路径（如 **`.cursor/req-docs/xxx.md`** 或 `stock-docs/需求_终稿.md`）。**可选**：用户补充的需求条件（范围、必须/禁止技术、与现有模块边界、性能/安全要求等）。 |
| **规范来源** | 优先参考 `template/后端技术模版.md`（Cursor 下即 `.cursor/template/后端技术模版.md`）；输出章节按需选用，不必全有。                                                                                       |
| **行为**     | 读取需求与条件 → 加载项目 Rules、Skills 及已有技术方案 → 对齐项目约定（接口、配置、QMQ、错误码、表设计等）→ 撰写后端技术文档。                                                |
| **输出**     | 默认 **`req-docs/<方案名>_技术方案.md`**（如 `.cursor/req-docs/...`）。技术方案**供后续代码实现使用**，不用于生成 Rules/Skills。                                                           |

**输出章节（按需选用）**：需求概述、重点问题概述、外部接口对接、配置、QMQ、接口及流程说明、关联接口调用流程、错误码、表设计。与需求无关的章节可整节省略。

---

以下命令按**推荐顺序**使用：**/genStructureDoc** → **/spec2context-md** → **/generateProjectContext**。另有配合使用的 **/deleteProjectContext**、**/pdf4code-md**。

### 2.1 /genStructureDoc

根据用户说明、已有文档或扫描代码，生成**项目架构说明初稿**（无固定格式，以描述清楚为准）。

| 项目     | 说明                                                                                                                                                                                                                                                       |
| -------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **入参** | 可选。**第一参数**：一段纯文字说明，或**文档路径**（如 `README.md`、`stock-docs/xxx.md`）；不传则进入「无输入」流程（扫描代码，需用户确认，不保证质量）。**第二参数**：输出路径（默认 `stock-docs/<项目名>架构说明_初稿.md`）。                        |
| **行为** | 有输入时：读取说明或文档 → 结合项目目录与关键文件补充、归纳 → 生成架构说明初稿；若说明较宽泛则引导用户补充代码路径、模块划分、入口等。无输入时：先提示「是否确认不传递参数，仍使用 AI 扫描代码生成？（不保证质量）」，仅当用户明确确认后才执行扫描与生成。 |
| **输出** | 默认 `stock-docs/<项目名>架构说明_初稿.md`；可再通过 `/spec2context-md` 转为规范格式终稿，并配合 `/generateProjectContext` 生成 Rules、Skills。                                                                                                          |

### 2.2 /spec2context-md

将 PDF 或 Markdown 转为《终稿模版》规范格式的 MD。

| 项目         | 说明                                                                                                                                                                                                      |
| ------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **入参**     | 一个：**PDF 路径**或 **MD 路径**；可选第二个参数为输出路径                                                                                                                                                |
| **规范来源** | 优先读取 `template/终稿模版.md`；若不存在则用命令内嵌结构                                                                                                                               |
| **输出位置** | PDF 首次：`stock-docs/<方案名>_初稿.md`（初稿）；MD/初稿转为规范格式：`stock-docs/<方案名>_终稿.md`（终稿）。详见 [文档产物阶段](./README-目录与路径约定.md#3-文档产物阶段原稿--初稿--终稿)。 |

传入 Markdown 时：分析并重组为「核心概念、业务规则、关键流程」等模板结构，写入终稿。传入 PDF 时：首次执行输出初稿，用户确认后再执行一次（传入初稿路径）输出终稿。完成后提示可执行 `/generateProjectContext` 并传入 `stock-docs/<方案名>_终稿.md`。

### 2.3 /generateProjectContext

根据文档生成 Rules、Skills、文档索引（及按需更新 main.mdc）。

| 项目         | 说明                                                                                                                                                                                                                                                                                                                                                                                                                 |
| ------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **入参**     | 一个：**URL**（如 `https://xxx.com/doc`）或 **本地路径**（如 `stock-docs/拼团技术方案设计.md`）                                                                                                                                                                                                                                                                                                                    |
| **文档位置** | 本命令入参的本地文档须为 **`stock-docs/`** 下的存量源文档（与 **`req-docs/`** 中的技术方案目录分工不同）                                                                                                                                                                                                                                                                                                                                                                                 |
| **行为**     | 大模型拉取/读取文档 → 分析（概念、状态、流程、规则）→ 按「拆解与分工」生成/更新 main.mdc（若适用）、专题 Rules、专题 Skills、docs-index 一行                                                                                                                                                                                                                                                                         |
| **输出**     | `rules/main.mdc`（若为功能模块或公共模块）、`rules/<主题>-context.mdc`、`skills/<主题>/SKILL.md`、`docs-index.md` 中该文档对应行；若入参为 URL 还会把原文保存到 `stock-docs/<主题>.md`。**入参通常为终稿文档**（如 `stock-docs/<方案名>_终稿.md`，相对配置根），但生成的 Rules、Skills **不带 `_终稿` 后缀**，见 [文档产物阶段](./README-目录与路径约定.md#3-文档产物阶段原稿--初稿--终稿)。 |

**注意**：同一文档多次执行会**更新**该文档对应的 Rules、Skills 及 docs-index 该行，不会重复追加行。内网 URL 可能无法访问，请先将内容保存到 `stock-docs/xxx.md` 再传路径。

### 2.4 /deleteProjectContext

删除某份文档对应的 Rules、Skills、索引行及 main 中的相关描述。

| 项目     | 说明                                                                                                                                                         |
| -------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **入参** | 一个：**文档路径**（如 `stock-docs/拼团技术方案设计.md`）或 **可匹配的片段**（如 `拼团技术方案设计.md`）                                                   |
| **行为** | 在 docs-index 中匹配该文档行 → 删除该行中的 Rules、Skills 文件及 Skill 空目录 → 从 docs-index 删除该行 → 更新 main.mdc（删除模块一览对应行或公共能力入口节） |
| **结果** | 该文档不再拥有对应 Rule/Skill，索引与 main 中相关描述被移除；**`stock-docs/`** 下的源文档不删除                                                              |

**注意**：若匹配到多行或删除范围不明确，大模型会向用户确认后再执行。

### 2.5 /pdf4code-md

将 PDF 技术方案文档转为 Markdown 并保存到项目，并可引导用户补全流程说明（流程图无法从 PDF 解析时）。

| 项目         | 说明                                                                                                                                                                   |
| ------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **入参**     | 一个：**PDF 技术方案文档的本地路径**                                                                                                                                   |
| **行为**     | 读取 PDF → 提取文字并转为 Markdown → 保存到项目（技术方案推荐 `req-docs/<方案名>.md`；若仅作上下文源文档可放 `stock-docs/<方案名>.md`）；可选：引导用户以图片或文字提供流程图/流程说明，并追加到该 MD |
| **输出**     | 保存后的 MD 路径；若用户补全流程说明则写入该 MD                                                                                                                        |
| **典型用法** | 在「提问与实现环节」之前，若技术方案仅为 PDF，可先执行本命令转成 MD，再执行 `/opsx-new` 等。                                                                           |

## 3. 阶段二：提问与实现、实现后

### 3.1 OpenSpec 工作流

- **/opsx-new**：提问并新建变更，按 artifact 填写 proposal 等。
- **/opsx-continue**：继续根据提案生成剩余产物（specs、design、tasks）。
- **/opsx-apply**：根据规划（tasks）生成代码。
- **/opsx-archive**、**/opsx-ff**、**/opsx-explore**、**/opsx-onboard**、**/opsx-sync**、**/opsx-verify**、**/opsx-bulk-archive**：变更归档、快进、探索、接入引导、同步规范、校验、批量归档等。

详见 [OpenSpec-介绍](./OpenSpec-介绍.md) 与 [Flow2Spec使用说明](./Flow2Spec使用说明.md)。

### 3.2 global-fix / global-feat / global-sync

| 命令                                         | 何时用                                       | 说明                                                                                                                                                                                          |
| -------------------------------------------- | -------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **/global-fix**（/修正实现规则）             | 实现后用户指出某处违反项目规则或约定         | 根据用户描述修正相关代码，并同步更新与该约定相关的文档、`rules/*.mdc` 与 `skills/*/SKILL.md`，必要时更新变更产物。详见 `commands/global-fix.md`。 |
| **/global-feat**（/新增能力）                | 用户要新增能力（功能、模块或约定）           | 若尚未实现则按项目约定补全实现并补充文档、规则与 Skill；若已实现则基于现有实现与规则仅补充/对齐文档与 Rule/Skill。详见 `commands/global-feat.md`。                                    |
| **/global-sync**                             | 需要把方案沉淀到全局并同步规范时             | 一条命令完成「技术方案 → 功能概述 → 提交到全局 Rules/Skills → 同步规范到 openspec/specs」；可传技术方案路径或变更名。                                                                         |
| **/global-merge-context**（/合并上下文冲突） | merge / rebase 后仍存在 `<<<<<<<` 等冲突标记 | 上下文类（索引、规则、技能、说明文档）自动合并；实现与对外配置等不擅自合并。详见 [§3.3](#33-global-merge-context) 与 `commands/global-merge-context.md`。                             |

### 3.3 global-merge-context

中文斜杠名：**/合并上下文冲突**。在 **merge / rebase** 后仍存在 Git 冲突标记（`<<<<<<<` / `=======` / `>>>>>>>`）时使用。与 **global-fix**（用户已指明问题后的定向修正）不同：本命令处理**合并产生的批量冲突**，并强制区分「可自动合并的编辑器上下文」与「须用户确认的实现侧」。

| 项目             | 说明                                                                                                                                                                                                                                                                                                |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **入参**         | **可选**。不传：在工作区内检索仍含冲突标记的文件并分类处理。传参：指定一个或多个仍含冲突的文件（`@` 引用或路径列表），优先只处理这些文件；完成后可询问是否对工作区补充扫描。                                                                                                                        |
| **可自动合并**   | 文档索引（如 `docs-index.md`）、总览/模块规则（`rules/`）、技能（`skills/`）、与规则/技能配套的说明类 Markdown、索引联动的纯说明文档等；按「并集、去重、不丢任一侧独有约束」等策略合并，**必须删净冲突标记**。细则见 `commands/global-merge-context.md`。 |
| **禁止自动合并** | 应用/服务实现源码、会改变对外行为的配置（路由、函数注册、中间件链等）、依赖与锁文件、构建脚本，以及集中注册且两侧条目不一致的外部资源清单等；**只输出差异摘要与建议，待用户选定后再改**。                                                                                                           |
| **输出**         | Markdown 摘要：已解决文件与要点；待确认文件与两侧差异及建议；确认无残留冲突标记（未全量扫描时可提示补充扫描）。                                                                                                                                                                                     |

**典型场景**：两个分支都修改了 **docs-index、main.mdc、同一专题的 rules/skills 或 `stock-docs/` 下说明** 时，优先用本命令对齐上下文，减少漏行、漏规则。

## 4. 推荐执行顺序（Flow2Spec）

### 需求与方案（可选）

当有 PRD 或需求需先澄清再写技术方案时：

| 顺序 | 命令                     | 作用                                                           |
| ---- | ------------------------ | -------------------------------------------------------------- |
| 1    | **/需求澄清**             | 针对 PRD/需求反问，直到需求完全清楚                            |
| 2    | **/生成后端技术文档**     | 根据澄清后需求（+ 可选需求条件）基于项目知识库生成后端技术文档 |

### 上下文生成

| 顺序 | 命令                        | 作用                                         |
| ---- | --------------------------- | -------------------------------------------- |
| 1    | **/genStructureDoc**        | 生成项目架构说明**初稿**                     |
| 2    | **/spec2context-md**        | 将初稿转为《终稿模版》规范格式，得到**终稿** |
| 3    | **/generateProjectContext** | 根据终稿生成 **Rules、Skills、文档索引**     |

### 提问与实现环节

若技术方案仅为 **PDF**，可先执行 **`/pdf4code-md <PDF路径>`** 转成 MD，再进入下列步骤。

| 顺序 | 命令                      | 作用                                                                                         |
| ---- | ------------------------- | -------------------------------------------------------------------------------------------- |
| 1    | **/opsx-new**             | **提问**并新建变更，按 artifact 填写 proposal 等                                             |
| 2    | **/opsx-continue**        | **继续**根据提案生成剩余产物（specs、design、tasks）                                         |
| 3    | **/opsx-apply**           | **根据规划**（tasks）生成代码                                                                |
| 4    | **/global-fix**           | **实现后**用户指出规则错误时：修正代码并同步更新文档与全局 Rules/Skills                      |
| —    | **/global-feat**          | **新增能力**时：补全实现与文档，或已实现则仅补文档与规则                                     |
| 5    | **/global-sync**          | 生成全局 Skills、Rules 并**归档**（可选）                                                    |
| —    | **/global-merge-context** | **merge/rebase 后**：自动合并索引/规则/技能/说明类冲突；实现与配置类仅对比与建议，待用户确认 |

完成「上下文生成」1～3 后，再按「提问与实现环节」1～5 进行变更留档与代码实现；**global-fix** 在实现后、用户指出某处违反规则时使用；**global-feat** 在新增能力时使用；**global-sync** 可按需执行。**global-merge-context** 在合并分支后出现冲突标记时使用，与上述命令无固定先后顺序。

## 6. 命令关系简图

```
flow2spec init [agent ...]
  └── 创建 配置根/*（可多套）、openspec/；配置根为 .cursor 时注册 Cursor 斜杠命令

推荐顺序（需求与方案）：需求澄清 → 后端方案
推荐顺序（上下文）：genStructureDoc → spec2context-md → generateProjectContext
推荐顺序（提问与实现）：（可选 pdf4code-md）→ opsx-new → opsx-continue → opsx-apply → global-sync（可选）

需求澄清             └── PRD/需求 → 反问直到需求完全清楚
生成后端技术文档     └── 澄清后需求（+ 需求条件）→ req-docs/<方案名>_技术方案.md（基于项目知识库/Skills/Rules）
genStructureDoc      └── 说明/文档（或扫描）→ stock-docs/<项目名>架构说明_初稿.md
spec2context-md      └── PDF/MD → 初稿/终稿 → stock-docs/
generateProjectContext └── stock-docs/xxx.md（或 URL）→ main + Rules + Skills + docs-index
deleteProjectContext  └── 文档路径/片段 → 删该文档对应的 Rules、Skills、索引行、main 相关描述
pdf4code-md         └── PDF 路径 → 转 Markdown 并保存到 req-docs/<方案名>.md，可补流程说明

opsx-new / opsx-continue / opsx-apply / opsx-archive 等 └── OpenSpec 变更留档与实现
global-fix          └── 用户指出规则错误 → 修正代码 + 更新文档与 Rules/Skills（全局工作流，不归纳到 opsx）
global-feat         └── 新增能力 → 补全实现（若未实现）+ 文档 / Rules / Skill；已实现则仅补文档与规则
global-sync         └── 方案→概述→全局 Rules/Skills→同步规范（可选）
global-merge-context └── merge/rebase 冲突 → 上下文类自动合并；实现/配置类仅对比与建议，待用户确认
```

## 7. 快速参考（按阶段）

| 阶段           | 想做的事                                                             | 用哪个命令                                                                                     |
| -------------- | -------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------- |
| **项目首次**   | 在当前项目使用 Flow2Spec                                             | `flow2spec init` 或 `flow2spec init claude` 等；见 `flow2spec --help`                                                |
| **需求与方案** | 针对 PRD/需求反问直到清楚                                            | `/需求澄清`：可传 PRD 全文、需求描述或文档路径                                      |
|                | 根据澄清后需求生成后端技术文档                                       | `/生成后端技术文档`：必传需求或文档路径，可附带需求条件；输出默认 `req-docs/<方案名>_技术方案.md` |
| **上下文生成** | 生成项目架构说明初稿                                                 | `/genStructureDoc`：可传说明文字或文档路径；输出默认 `stock-docs/<项目名>架构说明_初稿.md`   |
|                | 把 PDF 或杂乱 MD 变成规范技术方案（终稿）                            | `/spec2context-md` + `stock-docs/xxx.pdf` 或 `xxx.md`；PDF 先出初稿，再执行一次出终稿            |
|                | 把需求/方案文档变成 AI 可用的 Rules、Skills                          | `/generateProjectContext` + `stock-docs/xxx.md`（通常 `_终稿.md`）                   |
|                | 不再需要某文档的上下文                                               | `/deleteProjectContext` + `stock-docs/xxx.md` 或 `xxx`                                           |
|                | 更新某文档的 Rules、Skills                                           | 改文档后再次 `/generateProjectContext` + 同一路径                                     |
| **提问与实现** | 手头只有 PDF、要进入提问与实现环节                                   | 先 `/pdf4code-md <PDF路径>` 转成 MD，再 `/opsx-new` 等                                         |
|                | 提问并新建变更                                                       | `/opsx-new <变更名>`                                                                           |
|                | 继续根据提案生成剩余产物                                             | `/opsx-continue`                                                                               |
|                | 根据规划（tasks）生成代码                                            | `/opsx-apply`                                                                                  |
| **实现后**     | 用户指出规则错误：改代码并同步文档与规则                             | `/global-fix`（或 `/修正实现规则`）                                                            |
|                | 新增能力（功能/模块/约定）：补全实现与文档，或已实现则仅补文档与规则 | `/新增能力`（或 `/global-feat`）                                                               |
|                | 生成全局 Skills、Rules 并归档                                        | `/global-sync`（可选）                                                                         |
|                | merge/rebase 后：规则/索引/技能/说明与代码同时有冲突                 | `/global-merge-context`（或 `/合并上下文冲突`）；可 `@` 指定冲突文件                           |
