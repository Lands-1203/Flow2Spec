---
name: /修正实现规则
id: global-fix
category: 工作流
description: 根据用户指出的实现规则错误，修正代码并同步更新文档与全局规则
---

在代码实现后，若用户指出某处违反了项目规则或约定，则修正实现并同步更新相关文档与全局规则（及可选的变更/设计产物），使该约定被明确记录、后续实现与 AI 可遵循。

**输入**：用户描述「哪里错了、应遵循什么规则」。例如：
- 「某处违反了 XX 规则，请改掉并补充到规则里」
- 「Y 模块下所有 Z 的写法错了，应该是…」
- 「只改某目录 / 某类文件」

用户可附带**范围**（如具体目录、文件或「全项目同类写法」）；若项目使用变更管理，可附带变更名。

**步骤**

1. **解析用户指出的问题**
   - 明确违反的**规则或约定**（正确写法 vs 当前错误写法）。
   - 明确**范围**：具体文件、目录或「全项目同类写法」。
   - 若范围不清，可询问用户后再继续。

2. **修正代码**
   - 在范围内按正确写法修改所有相关位置（编辑文件）。
   - 若用户表达为「所有这类」或规则为全局性，可搜索代码库中同类错误写法并一并修正。

3. **更新相关文档**
   - 判断与该约定相关的文档（项目约定的文档目录，如 `.cursor/docs`、`docs` 等；若使用变更管理则含变更目录下的 design/spec）。
   - 在文档中新增或修订一小节，写清该约定，必要时注明对应 rule/skill。

4. **更新全局规则与 Skill**
   - 若该约定为项目级或模块级：
     - 在项目约定的规则目录（如 `.cursor/rules/`）下新增或更新一条规则，写清「必须/禁止」及适用范围（globs 等）。
     - 在项目约定的技能目录（如 `.cursor/skills/`）下新增或更新一个 Skill，写清触发词与正确/错误示例，便于后续实现与 AI 按规则执行。
   - 规则与 Skill 中可引用文档作为约定出处。

5. **（可选）更新变更/设计产物**
   - 若项目使用 OpenSpec 或类似变更管理，且该约定应纳入某次变更的设计或规格：更新对应 design 或 spec，写明该规则，便于后续实现与校验一致。

6. **输出摘要**
   - **代码**：修改了哪些文件、改了什么。
   - **文档**：更新了哪些路径、新增/修订了哪一节。
   - **规则/Skill**：新增或更新了哪条 rule、哪个 skill。
   - 若更新了变更/设计产物，一并说明。

**输出格式示例**

```markdown
## 修正：<规则简述>

### 代码
- <路径>：<具体修改说明>
- …

### 文档
- <文档路径>：新增/修订「…」小节
- …

### 规则 / Skill
- <规则路径>：新增/更新 — …
- <Skill 路径>：新增/更新 — 触发词：…

（可选）变更/设计产物：<路径> — …
```

**约束**
- 范围不明时：按最可能范围（如用户提到的模块或近期改动的文件）修正，并说明「若其他位置也有同类写法，可再说，我一起改」。
- 至少更新「文档、规则、Skill」中的一类，使约定被书面化。
- 一条约定优先对应一条规则 + 一个 Skill；若已有合适 rule/skill 文件，在其上增补即可。

**何时使用**
- 实现完成后，用户指出「这里违反规则」「写错了」「应该按 XX 约定来」时。
- 用户希望既改代码，又把规则沉淀到文档与 Cursor（rules/skills）以便以后统一遵循时。
