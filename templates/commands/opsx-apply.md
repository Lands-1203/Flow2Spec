按 OpenSpec 变更中的任务进行实现。

**输入**：可选的变更名称（如 `/opsx:apply add-auth`）。若未提供，尝试从对话上下文推断；若模糊或歧义，必须让用户从可用变更中选择。

**步骤**

1. **选定变更**

   若提供了名称则使用。否则：
   - 若用户提到过某变更，从对话推断
   - 若仅有一个活跃变更，可自动选中
   - 若有歧义，执行 `openspec list --json` 获取变更列表，用 **AskUserQuestion 工具**让用户选择

   始终说明：「当前变更：<name>」，以及如何覆盖（如 `/opsx:apply <其他名称>`）。

2. **查看状态以确认 schema**
   ```bash
   openspec status --change "<name>" --json
   ```
   解析 JSON 得到：
   - `schemaName`：当前工作流（如 "spec-driven"）
   - 哪个 artifact 包含任务（spec-driven 下多为 "tasks"，其他 schema 看 status）

3. **获取 apply 说明**

   ```bash
   openspec instructions apply --change "<name>" --json
   ```

   返回内容包括：
   - 上下文文件路径（因 schema 而异）
   - 进度（总数、已完成、剩余）
   - 任务列表及状态
   - 基于当前状态的动态说明

   **状态处理：**
   - 若 `state: "blocked"`（缺少 artifact）：提示并建议先执行 `/opsx:continue`
   - 若 `state: "all_done"`：祝贺并建议归档
   - 否则：继续实现

4. **读取上下文文件**

   按 apply 说明中的 `contextFiles` 读取列出的文件。
   具体文件因 schema 而异：
   - **spec-driven**：proposal、specs、design、tasks
   - 其他 schema：按 CLI 返回的 contextFiles 处理

5. **展示当前进度**

   显示：
   - 使用的 schema
   - 进度：「N/M 任务已完成」
   - 剩余任务概览
   - CLI 返回的动态说明

6. **按任务实现（循环直到完成或阻塞）**

   对每个待办任务：
   - 标明正在做的任务
   - 完成所需代码修改
   - 修改尽量小、聚焦
   - 在 tasks 文件中将任务标为完成：`- [ ]` → `- [x]`
   - 继续下一项

   **以下情况暂停：**
   - 任务不清晰 → 请求澄清
   - 实现暴露出设计问题 → 建议更新 artifact
   - 遇到错误或阻塞 → 汇报并等待指示
   - 用户打断

7. **完成或暂停时展示状态**

   显示：
   - 本轮完成的任务
   - 总进度：「N/M 任务已完成」
   - 若全部完成：建议归档
   - 若暂停：说明原因并等待指示

**实现过程中的输出示例**

```
## 正在实现：<change-name>（schema: <schema-name>）

正在做任务 3/7：<任务描述>
[...实现中...]
✓ 任务完成

正在做任务 4/7：<任务描述>
[...实现中...]
✓ 任务完成
```

**完成时的输出示例**

```
## 实现完成

**变更：** <change-name>
**Schema：** <schema-name>
**进度：** 7/7 任务已完成 ✓

### 本轮完成
- [x] 任务 1
- [x] 任务 2
...

全部任务已完成！可使用 `/opsx:archive` 归档此变更。
```

**暂停时（遇到问题）的输出示例**

```
## 实现已暂停

**变更：** <change-name>
**Schema：** <schema-name>
**进度：** 4/7 任务已完成

### 遇到的问题
<问题描述>

**可选操作：**
1. <选项 1>
2. <选项 2>
3. 其他方式

你希望怎么做？
```

**约束**
- 按任务顺序推进直到完成或阻塞
- 开始前务必先读上下文文件（来自 apply 说明）
- 任务不明确时先暂停询问再实现
- 若实现暴露出问题，暂停并建议更新 artifact
- 代码修改保持最小、限定在单任务范围
- 每完成一项任务立即更新任务勾选
- 遇错误、阻塞或需求不清时暂停，不要猜测
- 以 CLI 返回的 contextFiles 为准，不要假设具体文件名

**与工作流的衔接**

本命令支持「对某变更执行操作」的模式：

- **可随时调用**：在所有 artifact 完成前（只要已有 tasks）、部分实现后、或与其他操作交替进行
- **允许更新 artifact**：若实现暴露出设计问题，可建议更新 artifact，不锁阶段，灵活推进
