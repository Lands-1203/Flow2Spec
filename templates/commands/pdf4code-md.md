# 将 PDF 技术方案文档转为 Markdown（并补全流程说明）

用户会在本命令后附带**一个参数**：**PDF 技术方案文档的本地路径**（如 `~/Downloads/技术方案.pdf`、`docs/设计文档.pdf`）。请按以下步骤执行，将 PDF 技术方案文档转为 Markdown 并保存到项目根目录下，必要时引导用户补全流程说明。

---

## 步骤 1：读取 PDF 并转为 Markdown

1. **读取**用户传入的 PDF 文件，提取其中的**文字内容**（表格、章节、列表、代码块等尽量保留结构），整理为 Markdown 格式。
2. **保存到项目根目录**下，推荐路径：`**docs/<方案名>.md**`（技术方案用于「按方案实现代码」时，应放在 `**docs/**`；`.cursor/docs/` 用于放生成 Rules/Skills 的源文档，不用于实现）。文件名为原 PDF 文件名去掉 `.pdf` 后加 `.md`。
3. 若项目根下无对应目录，可先创建再写入。
4. 保存后告知用户：「已将该 PDF 转为 Markdown 并保存为 `xxx.md`。」

---

## 步骤 2：向用户提问获取流程图（可选但推荐）

PDF 内嵌的**流程图**无法被直接解析为步骤与分支，若需按图实现代码，需用户配合提供。

1. 向用户说明：「文档中可能包含流程图，我无法从 PDF 中解析图中的步骤与分支。若您后续会根据技术方案实现代码（见 .cursor/rules/implement-tech-design.mdc），建议补全流程说明：
  - **方式一**：将相关流程图以**图片形式**发到本次对话中，我将解析后以文字形式写入上述 MD；  
  - **方式二**：直接以**文字描述**每个接口/流程的步骤（如：1. 是否登录 2. 查某表 3. 判断某字段 → 返回结果），我将原样写入上述 MD。  
   若文档无流程图或暂不提供，可回复「跳过」，我将结束本命令。」
2. **若用户回复「跳过」或明确表示无需流程说明**：告知用户「在对话中提供上述 MD 路径，并说明要按该技术方案实现代码，我将按 .cursor/rules/implement-tech-design.mdc 执行。」并提示：「在 Flow2Spec 项目中，也可使用 **/opsx-new**（默认推荐，逐步提问并填写 proposal 等）或 **/opsx-ff**（快进建齐 artifact）继续实现技术方案，再配合 /opsx-apply 按任务生成代码。」并结束。
3. **若用户提供流程图（图片或文字）**：进入步骤 3。

---

## 步骤 3：将流程说明写入该 MD

1. 若用户提供的是**图片**：解析图片中的步骤、判断分支与返回，整理为文字步骤。
2. 若用户提供的是**文字**：直接采用。
3. 在该 MD 文件末尾（或新增「流程说明」章节）**追加**流程内容，格式示例：

```markdown
## 流程说明（由用户提供 / 由流程图解析）

### 示例接口 A
1. 前端发起请求
2. 后端：查询某表最后一条记录
3. 判断：是否有某 ID？是 → 返回 true，否 → 返回 false
4. 返回结果

### 示例接口 B
1. 是否登录 → 否 返回 401
2. 是否过期 → 是 返回 403
…
```

1. 保存后告知用户：「流程说明已写入 `xxx.md`。接下来可任选其一继续：在对话中提供该 MD 路径并说明要按技术方案实现代码，我将按 .cursor/rules/implement-tech-design.mdc 执行；或在 Flow2Spec 项目中使用 **/opsx-new**（默认推荐，逐步提问并填写 proposal 等）或 **/opsx-ff**（快进建齐 artifact）继续实现技术方案，再配合 /opsx-apply 按任务生成代码。」

---

## 约束与小结

- **路径**：用户传入的 PDF 路径可为绝对路径或相对于项目根的路径。输出 MD 建议保存在 `**docs/<方案名>.md**`（`docs/` 用于放需要实现成代码的文档；`.cursor/docs/` 用于放生成 Rules/Skills 的文档）。
- **本命令仅负责**：PDF → Markdown 转换 + 可选流程说明补全；不执行代码实现。完成后可提示用户：使用 **/opsx-new**（默认推荐）或 **/opsx-ff** 继续实现技术方案，再配合 /opsx-apply 生成代码；或直接在对话中提供生成的 MD 路径并说明按技术方案实现，AI 将按 implement-tech-design.mdc 执行。
