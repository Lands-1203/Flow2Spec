# flow2spec 架构说明初稿

> 本初稿由扫描项目结构生成，建议结合业务说明与代码细节进一步补充。

---

## 项目定位

`@ctrip/flow2spec` 是一个 CLI 工具兼模板分发系统，用于在目标业务代码仓库中初始化"文档前置 AI 工作流"。核心思路是：把**规则（rules）**、**技能（skills）**、**文档模版（template）**和**知识库目录（stock-docs / req-docs）**一次性写入目标仓库的 AI 工具配置目录（`.cursor/`、`.claude/`、`.codex/` 等），形成**知识库 ↔ 实现 ↔ 知识库**的闭环，让 AI Agent 在对话中优先依据已沉淀的知识库而非零散聊天记录工作。

---

## 技术栈

- **运行时**：Node.js ≥ 16，纯 CommonJS，无额外运行时依赖
- **发布方式**：npm 包（`npx @ctrip/flow2spec init`），`bin` 字段指向 `cli.js`
- **核心操作**：文件系统递归复制 + 简单正则替换（无构建步骤）

## 目录结构

```
flow2spec/
├── cli.js                   # 命令行入口，解析参数，调用 lib/init.js
├── package.json
├── README.md
├── lib/                     # 核心逻辑
│   ├── agents.js            # Agent 定义（cursor / claude / codex）及 ID 校验
│   ├── init.js              # 初始化主流程：建目录、复制模板、触发格式转换
│   └── claudeRulesAdapter.js # .mdc → .md 格式适配（Cursor → Claude Code）
├── templates/               # 被分发到业务仓库的模板源
│   ├── rules/               # 规则文件（.mdc，init 时按 agent 做格式转换）
│   ├── skills/              # f2s-* 技能定义（SKILL.md，共 13 个子目录）
│   └── template/            # 文档输出模板（终稿模版.md、后端技术模版.md）
├── docs/                    # flow2spec 自身的使用文档（不分发）
└── scripts/                 # 暂为空
```

## 模块说明

### cli.js — 命令行入口

解析 `process.argv`，支持两条命令：
- `flow2spec --help`：打印帮助信息
- `flow2spec init [agent ...]`：调用 `lib/init.js`，将模板写入当前工作目录（`process.cwd()`）下对应 agent 的配置根

### lib/agents.js — Agent 注册表

定义三个支持的 AI 工具配置目录：

| Agent ID | 配置根     |
|----------|-----------|
| cursor   | `.cursor/` |
| claude   | `.claude/` |
| codex    | `.codex/`  |

并导出 `normalizeAgentIds`，负责参数校验与去重。`init` 不传 agent 时默认为 `cursor`。

### lib/init.js — 初始化主流程

执行步骤：
1. `ensureDirs`：在目标配置根下创建所有子目录（`rules/`、`skills/`、`template/`、`stock-docs/`、`req-docs/`）
2. `copyRulesTemplates`：复制 `templates/rules/`，按 agent 类型决定是否调用格式转换
3. 复制 `templates/skills/`（逐目录递归，已存在则覆盖）
4. 复制 `templates/template/`（文档输出模板）

所有路径以 `process.cwd()`（业务仓库根）为基准，`templates/` 取自 flow2spec 包自身的 `__dirname`。

### lib/claudeRulesAdapter.js — 格式适配

Cursor 规则格式（`.mdc`）与 Claude Code 规则格式（`.claude/rules/*.md`）存在差异：
- `globs:` → `paths:`（Claude Code 约定的路径范围键名）
- 移除 `alwaysApply: true/false`（Claude Code 不按此字段解析）
- 正文内 `.mdc` 文件引用改为 `.md`

仅当 `agentRoot === ".claude"` 时触发此转换。

## templates/ — 被分发的内容

### rules/（2 个规则文件）

| 文件 | 说明 |
|------|------|
| `implement-tech-design.mdc` | 引导 AI「按 req-docs 中的技术方案写代码」的规则 |
| `stock-docs-vs-req-docs.mdc` | 说明 stock-docs（存量上下文）与 req-docs（需求/方案）的职责划分 |

### skills/（13 个 f2s-* 技能）

技能以 `SKILL.md` 形式存放，AI Agent 在对话中按场景加载。按用途分组：

| 分组 | 技能 | 功能 |
|------|------|------|
| 文档生成链 | f2s-doc-arch | 生成架构说明初稿（本技能） |
| | f2s-doc-final | 将初稿规范化为《终稿模版》格式 |
| | f2s-doc-pdf | PDF 技术方案 → Markdown |
| | f2s-doc-add | 工作中把已落地能力（多文件）解析进知识库 |
| 上下文管理 | f2s-ctx-build | 终稿 → Rules / Skills / docs-index |
| | f2s-ctx-rm | 删除某文档对应的 Rules、Skills 及索引 |
| 需求侧 | f2s-req-clarify | 需求澄清（反问直到清楚） |
| | f2s-req-backend | 生成后端技术文档 |
| 知识库维护 | f2s-kb-feat | 新增能力并补全文档 |
| | f2s-kb-fix | 修正实现规则并同步文档 |
| | f2s-kb-sync | 会话/现状沉淀进知识库 |
| | f2s-kb-merge | 解决合并后编辑器上下文冲突 |
| 参考说明 | stock-docs-vs-req-docs | 解释目录分工（技能形式的说明文档） |

### template/（2 个文档模板）

- **终稿模版.md**：f2s-doc-final 的输出格式规范，后续 f2s-ctx-build 以此为输入
- **后端技术模版.md**：f2s-req-backend 输出的章节结构参考

## 关键路径与流程

### init 流程（单次、面向新仓库）

```
npx @ctrip/flow2spec init [agent]
    ↓
cli.js → lib/init.js
    ↓
在 {cwd}/{agentRoot}/ 下创建：
  rules/   ← templates/rules/ (按 agent 转换格式)
  skills/  ← templates/skills/
  template/ ← templates/template/
  stock-docs/  (空目录，存量上下文源)
  req-docs/    (空目录，需求与技术方案)
```

### 日常工作闭环（面向已初始化仓库）

```
[新能力/需求]
    ↓ f2s-req-clarify → f2s-req-backend → req-docs/ 技术方案
    ↓ implement-tech-design 规则驱动实现
    ↓ f2s-kb-feat / f2s-kb-fix（随时纠错扩展）
    ↓ f2s-kb-sync（阶段收尾，会话 → 知识库）

[架构沉淀]
    ↓ f2s-doc-arch → stock-docs/ 初稿
    ↓ f2s-doc-final → 终稿
    ↓ f2s-ctx-build → rules/ + skills/ + docs-index.md
```

## 配置与部署要点

- **发布**：标准 npm 发布，`files` 字段限定分发内容为 `cli.js`、`lib/`、`templates/`、`README.md`、`docs/`
- **无构建**：模板直接以源文件形式分发，init 时复制而非编译
- **覆盖策略**：`copyRecursive` 对已存在文件直接覆盖，每次 `flow2spec init` 可用于升级模板版本
- **格式转换幂等**：`.mdc → .md` 转换基于正则，对已是 `.md` 格式的文件重复执行无副作用
