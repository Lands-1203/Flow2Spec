# Flow2Spec × 市面同类 · 对比文案整合稿

> 把两轮反馈合并：**5 条能力维度**（纵轴）× **5 类真实对手**（横轴），然后给一段老实话。
> 用作 Slide 4 的蓝本——先看文案是否认同，再决定版式。

---

## 一句话定位

> Flow2Spec = 项目级「机读知识库」+ 多 IDE 原生加载 + 代码↔文档双向同步 + 任务进度留痕 + 缺口闸门。
> 不是新的 IDE、不是新的 agent、不是新的 spec 语言 —— 把**已有工具的连接处**补齐。

---

## 纵轴：5 条能力维度（你评一个方案合格与否，就看这 5 件事）

| 维度 | 说的是什么 | 不满足会怎样 |
|---|---|---|
| ① **多端复用** | 一份知识，Cursor / Claude / Codex / 未来新 IDE 都能用 | 换工具重写一份，团队分裂 |
| ② **路由收窄** | 一句话命中一个 topic，只读相关那 300 行 | 系统提示词扁平，token 爆、拿错段 |
| ③ **双向同步** | 改代码顺手改知识（`f2s-kb-feat/fix`），不是一次性生成 | 第二次迭代 spec 和 code 分家 |
| ④ **原生 agent** | 规则 + 知识喂给现有 agent，不劫持、不做 wrapper | 跟 IDE 抢主导权，新模型来了跟不上 |
| ⑤ **变更留痕 + 收尾闸门** | 改代码顺手改知识（`f2s-kb-feat/fix`），`.task/` 跨会话续作；**提交前 `f2s-git-commit` 强制三查：冲突扫 + KB 覆盖 + emoji/Conventional 规范**，不 `add -A`、不跳 hooks、不自动 push | 进度靠脑记，提交后才发现漏改知识 |

## 横轴：5 类真实对手 · 点名比

### 1. OpenSpec
- **它做得好**：spec 格式最严谨、有 change proposal、人读友好。
- **缺什么（对照纵轴）**：
  - ② 无机读路由：spec 是给人看的，AI 还是整段喂。
  - ③ **单向**：spec → code，code 漂移后不会反哺。
  - ⑤ 无任务进度机制。
- **一句话**：格式王者，但 spec 本身不会追代码的变化。

### 2. Superpowers / ohMyClaude / awesome-cursorrules 类技能仓
- **它做得好**：现成 subagent / skill / rule 一堆，即插即用。
- **缺什么**：
  - ① **绑一家 IDE**（Superpowers → Claude Code；awesome-cursorrules → Cursor）。
  - ② 无项目级知识层，只有通用提示词。
  - ③ 知识不随项目演化。
- **一句话**：好用的工具箱，但工具箱里没有**你这个项目**的地图。

### 3. 向量知识库 / RAG / embeddings
- **它做得好**：全文档索引，让 AI 自己捞段。
- **缺什么**：
  - ② **概率检索 ≠ 路由**，相似不等于正确。
  - 无依赖链、无硬约束、拿错段**静默失败**。
  - ⑤ 没有任务状态的概念。
- **一句话**：适合"搜一搜"，不适合"改代码必须先读清哪几条约束"。

### 4. Spec Kit / Kiro / BMAD（Spec-Driven Dev 类）
- **它做得好**：PM 友好，一次从需求直出 spec + 初代码。
- **缺什么**：
  - ③ **一次性**：生成即死，第二次迭代 spec 和 code 分家。
  - ② 无路由，spec 越写越长、越读越慢。
  - ④ 部分方案是 wrapper，跟 IDE 抢主导权。
- **一句话**：适合从 0 到 1，不适合从 1 到 N。

### 5. Claude Memory / 模型自带记忆
- **它做得好**：省得反复自我介绍。
- **缺什么**：
  - ① 绑模型。
  - ⑤ 不可版本化、团队不能共享、不可审阅。
- **一句话**：个人偏好可以用，项目知识别碰。

## Flow2Spec 自己的位置

不是上面任何一家的替代，而是**把它们没做的连接处补齐**：

| 维度 | Flow2Spec 怎么做 |
|---|---|
| ① 多端复用 | `.Knowledge/` 是事实源，`.cursor/.claude/.codex/` 各自按原生方式挂载规则 |
| ② 路由收窄 | `manifest-routing.json` + `matchers/*.json` 机读命中 → 依赖链展开 → **缺口闸门**强制澄清 |
| ③ 双向同步 | `f2s-kb-feat` / `f2s-kb-fix` 强制「改代码带着改知识」 |
| ④ 原生 agent | 只提供 rules + skills + knowledge，执行仍走 Cursor / Claude / Codex 原生 |
| ⑤ 变更留痕 + 收尾闸门 | `.task/todo.json` + `task.md` + `user-todos.md`，跨会话可续作；`f2s-git-commit` 提交前强制「冲突扫 / KB 覆盖查 / emoji+Conventional 首行」三查 |

---

## 老实话（必须讲，否则不可信）

> **Flow2Spec 也会腐化 —— 停用 `f2s-kb-feat` / `f2s-kb-fix`，知识库一样会偏离代码。**
>
> 差别在于：**腐化会被缺口闸门「喊」出来**，AI 读不懂就会停下要文档，不是静默拿错段继续改。
>
> —— 工具不能替代纪律，`f2s-*` 只是把同步的成本降到最低。

其他短板，也写清楚：
- **概念曲线陡**：manifest / matcher / topic / task 四个名词，不背半小时用不起来。
- **价值延迟**：单次任务看不出区别，两周后回头才有感（第 2 次类似需求时）。
- **中文冷启动**：英文生态起步慢，推广要自己造势。
